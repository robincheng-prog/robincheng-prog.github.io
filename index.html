學習 Docker

學習 Docker 是現代軟體開發、維運與 DevOps 領域的核心技能。Docker 是一種輕量級的虛擬化技術（容器技術），可以將應用程式與其依賴的環境打包在一起，讓程式在任何電腦上都能擁有完全相同的運行環境，實現「一次建置，到處執行」。
以下為您整理出結構化的 Docker 學習路徑與核心概念，幫助您從零開始建立完整的知識架構。

🐳 第一階段：核心觀念理解（先備知識）

在動手寫 Code 之前，必須先理解 Docker 的三大核心元件與它和傳統虛擬機（VM）的差別。 [4]

  • 映像檔 (Image)： 唯讀的模板。它包含了應用程式的所有內容（程式碼、函式庫、設定檔等），可以把它想像成應用程式的「安裝檔」。
  • 容器 (Container)： 映像檔運行的實體。它就像是一個輕量級的獨立沙盒環境，具有自己的檔案系統、網路和行程，彼此完全隔離。
  • 倉庫 (Registry)： 集中存放映像檔的地方。官方提供了 Docker Hub，您可以在上面找到各種官方及社群打包好的映像檔（例如 Ubuntu、MySQL、Nginx 等）。
  • Docker 與 VM 的差異： VM 需要為每個應用程式安裝完整的客體作業系統 (OS)；而 Docker 容器直接共用主機的 OS 核心，因此啟動更快、資源佔用更少。

🚀 第二階段：動手實作（基礎指令與工作流程）

學習 Docker 最快的方法就是直接在終端機（Terminal）下指令操作。

  1. 安裝： 前往 Docker 官方網站下載並安裝 Docker Desktop（Windows/Mac）或透過指令安裝 Docker Engine（Linux）。
  2. Hello World： 打開終端機，執行第一個指令 docker run hello-world，測試 Docker 是否安裝成功。
  3. 基礎指令練習：
  • docker pull：從倉庫下載映像檔。
    • docker images：列出目前本機擁有的映像檔。
    • docker run：啟動一個容器（例如：docker run -d -p 80:80 nginx，這會啟動一個 Nginx 網頁伺服器）。
    • docker ps：查看目前正在運行的容器（加上 -a 參數可查看所有容器）。
    • docker stop / docker start：停止或啟動容器。
    • docker rm：刪除容器。

📦 第三階段：進階實戰（打造專屬環境）

當您熟悉基礎指令後，就可以開始學習如何將自己的專案容器化。

  • Dockerfile (映像檔定義檔)： 這是一個文字檔，裡面寫滿了一步一步的指令（例如基礎映像檔是什麼、要複製哪些檔案、需要安裝什麼套件等），用來建立自訂的 Image。
  • 資料持久化 (Volumes)： 容器停止或刪除後，裡面的資料也會跟著消失。此時需要學習 Volume，將容器內的資料掛載到本機硬碟上保存。
  • 網路設定 (Networking)： 學習如何讓多個 Docker 容器互相溝通，或者對外開放特定的埠口（Port）。

🛠️ 第四階段：多容器管理 (Docker Compose)

當您的專案不僅只有一個服務，而是包含前端、後端、資料庫等多個容器時，手動逐一啟動會非常複雜。

  • 學習撰寫 docker-compose.yml 檔案，只要一個指令 docker compose up，就能定義並啟動所有相關聯的容器服務。

⚠️ Linux 環境下的權限小陷阱

在 Linux 上使用 Volume 掛載時，初學者常遇到兩個常見問題：

  • 檔案權限衝突：Docker 容器預設以 root 身分執行。如果你的容器會在掛載目錄內「產生新檔案」（例如寫入 Log 或 SQLite 資料庫），那些檔案在本機的擁有者會變成 root，導致你在 Linux 本機無法修改它們。
  • 解決方式：在 docker run 時加入 --user $(id -u):$(id -g) 參數，讓容器內的 Python 改用你目前 Linux 帳號的 UID/GID 來執行。

NVIDIA OpenShell

NVIDIA OpenShell 是 NVIDIA 針對自主與自演化 AI 代理（Autonomous AI Agents）所打造的開源安全執行環境（Runtime）。它透過「沙箱隔離」、「政策強制引擎」與「隱私路由器」三層防禦機制，讓具備高權限的 AI 代理能在受控且安全的環境下運作，避免其在執行複雜任務或自我調適時帶來系統安全風險。

核心功能與架構

NVIDIA OpenShell 主要藉由以下三大核心模組來實現企業級的 AI 安全控管：

1. 沙箱隔離（Sandbox Isolation）

  • 核心機制：為 AI 代理建立一個與生產環境、主機資料庫與敏感基礎設施完全隔離的執行空間。
  • 實作方式：在底層使用容器與微虛擬機（Micro-VM）技術，結合 Linux 核心級別的防護（如 netns、seccomp BPF 與 Landlock LSM），嚴格限制 AI 代理的系統呼叫（System Calls）及網路訪問權限。
  • 優勢：確保即使 AI 代理遭遇提示詞注入（Prompt Injection）或生成了惡意程式碼，也能將其封鎖在受限環境內，無法對主機造成損害。

2. 政策引擎（Policy Engine）

  • 核心機制：採用 OPA（Open Policy Agent）政策即代碼（Policy-as-Code）策略，針對 AI 代理的行為設定明確的「允許」與「拒絕」規則。
  • 動態更新：支援即時的政策更新，可根據不同專案需求，靈活調整代理能存取的檔案路徑、API 呼叫或硬體資源。
  • 預設阻斷（Default Deny）：所有未被明確允許的系統或網路操作，系統預設一律攔截，確保最低權限原則（Principle of Least Privilege）。

3. 隱私路由器（Privacy Router）

  • 核心機制：扮演 AI 代理與外部世界的中介樞紐。
  • 資料過濾：在 AI 代理發送 API 請求或接收外部資料前，自動進行機敏資料（如 PII 個資、企業機密）的檢測、遮蔽或脫敏處理，防止資料外洩。

應用情境與生態系整合

  • NemoClaw 整合：OpenShell 是 NVIDIA 開源軟體堆疊（如 NemoClaw）的核心安全元件。開發者可以透過定義 Blueprint（藍圖）來規範代理的行為，將 Guardrails（安全護欄）的責任從 Prompt 提示詞層級轉移到 OpenShell 沙箱底層執行。
  • 本地端開發與企業佈署：無論是開發者在個人開發環境測試，還是企業級擴展到資料中心，OpenShell 皆提供一致的安全基元（Primitives）。

AI 運作方式 從 token 到 agent

AI（人工智慧）的運作是一個層層堆疊的過程。簡單來說，AI 將人類語言拆解為基本單位（Token），透過神經網絡進行理解與數學預測，最終透過外掛工具與自主規劃能力，晉升為能解決複雜任務的智慧代理（Agent）。
以下為您拆解從「Token」到「Agent」的完整細節：

1. Token（權杖/字詞片段）：AI 世界的最小單位

在 AI（特別是大型語言模型 LLM）的世界中，Token 是模型真正用來「思考」和「計算」的最小語意單位。AI 無法直接閱讀人類的文字（如中文字或英文字母），必須先透過 Tokenizer（分詞器） 將文本切割成 Token，再轉換成數字向量才能處理。
以下為您深入拆解 Token 的技術細節與運作機制：

1.1. 文字如何變成 Token？（以 Byte-Pair Encoding 為例）

現代大模型（如 ChatGPT、Claude）主要使用 BPE（字節對編碼） 演算法。它不是單純按「單字」或「字母」切割，而是尋找文本中最常出現的字詞片段（Subwords）。

  • 英文範例：
  • 常用詞會自成一個 Token，例如 the、cat。
    • 不常用或複雜的詞會被拆開，例如 unhappiness 可能被拆成 un + happi + ness（共 3 個 Token）。
  • 中文範例：
  • 中文的結構不同。常見的詞（如 電腦、人工智慧）或常用字（如 我、是）會自成一個 Token。
    • 罕見字或特殊組合可能被拆成單個字元，甚至退化成底層的「字節（Byte）」編碼。

1.2. 中英文的 Token 消耗差異（為什麼中文比較貴？）

因為 BPE 演算法最初多以英文語料為基礎設計，導致中文的 Token 轉換效率通常比英文低。

  • 英文：1 個單字平均約等於 1.3 個 Token。
  • 中文：1 個中文字平均需要 1 到 2 個 Token，有些模型（特別是早期模型）甚至一個中文字需要 2~3 個 Token。
  • 影響：在輸入相同長度的意思時，中文消耗的 Token 數量通常是英文的 2 到 3 倍。這也是為什麼中文使用者在付費 API 上成本較高的原因。

1.3. 從 Token 到數學：向量嵌入（Embedding）

當文字被切成 Token 後，模型會透過查表，將每個 Token 轉換成一串長度固定的浮點數，這叫做 Embedding（嵌入向量）。

  • 高維幾何空間：每個 Token 會變成一個擁有數千個維度的座標點（例如 1536 維）。
  • 語意接近度：在 composite 空間中，含義相近的 Token 距離會非常近。
  • 數學公式可以計算出：
    • 蘋果 + 科技 ~ iPhone
    • 國王 - 男人 + 女人 ~ 女王

1.4. 為什麼 Token 數量這麼重要？

在 AI 應用開發中，Token 限制了模型的兩大核心能力：

  • Context Window（上下文視窗）：模型單次記憶的上限（包含你輸入的提示詞 + AI 回應的字數）。如果超過 Token 限制，AI 就會「遺忘」最前面的對話。
  • 計費與速度：所有 LLM API（如 OpenAI, Anthropic）都是「按 Token 數量」計費。輸入與輸出的 Token 越多，費用越高，且模型生成下一個 Token 需要運算時間，Token 越多速度越慢。

2. 向量資料庫與 Context（上下文）：AI 的長期記憶

如果說 Token 是 AI 思考的「基本詞彙」，那麼 Context（上下文） 就是它的「工作記憶（短期記憶）」，而 向量資料庫（Vector Database） 則是它的「硬碟（長期記憶）」。
大型語言模型（LLM）每次對話就像「金魚的腦袋」，一旦視窗滿了就會遺忘。為了讓 AI 具備連續性、記住公司內部文件或使用者的偏好，我們必須仰賴上下文工程（Context Engineering）與向量資料庫的協同運作。
以下為您拆解 AI 「短期記憶」與「長期記憶」的運作細節：

2.1. Context Window：受限的短期工作記憶

Context（上下文視窗） 是 AI 執行當下任務時，能夠同時讀取的 Token 總量。

  • 運作機制：你輸入的 Prompt、歷史對話紀錄、以及 AI 剛才講過的話，都必須塞進這個視窗裡。
  • 致命限制：視窗寬度是有限的（例如 8K 到 128K Tokens）。一旦對話太長、或是塞入的文件太大，超過限制的部分就會被直接切掉（AI 出現「失憶」症狀）。
  • 成本昂貴：Context 屬於模型運算時的即時記憶，每次對話都要重新計算所有 Token 的注意力機制（Attention），因此 Context 塞得越滿，API 費用越貴、反應速度越慢。 [

2.2. 向量資料庫：突破限制的長期記憶庫

為了讓 AI 能存取數百萬字的公司文件或長期的對話歷史，我們需要把資料放在外部的向量資料庫中（例如 Pinecone、Qdrant、Weaviate）。

  • 高維向量化（Embedding）：將大篇幅的非結構化文字（如 PDF、技術手冊、過往對話）切成段落，並透過 Embedding 模型轉換成「向量數字」存入資料庫。
  • 語意相似度檢索（ANN Search）：當使用者問了一個問題，資料庫不用靠精準的「關鍵字」比對，而是利用數學公式（如餘弦相似度）找出「意思最相近」的幾段文字。

2.3. RAG（檢索增強生成）：長期與短期的記憶交換橋樑

RAG（Retrieval-Augmented Generation） 是將長期記憶提煉成即時上下文的標準架構：

透過這個流程，AI 不需要把整本百科全書背下來（塞進 Context），而是需要什麼答案時，去向量資料庫（長期記憶）「翻書」，再把重點撈回 Context（短期記憶）中進行精準回答。

2.4. 2026 年最新趨勢：三層記憶模型 (Memory-First)

隨著 AI Agent 的發展，業界發現單純使用「向量資料庫做 RAG」是不夠的。目前的 AI 記憶已經進化為結構化的三層記憶體系統：

記憶層級 對應技術 存放內容 特點
工作記憶 (Working) Context Window 當下對話、剛撈出來的資料片段 最精確、成本最高、關機即逝
情節記憶 (Episodic) 記憶檔案（如 memory.md）/ 向量庫 過去幾次對話摘要、使用者偏好風格 維持 Agent 行為與人設的一致性
語義記憶 (Semantic) 向量資料庫 / 知識圖譜 企業內部文件、法規、龐大的知識庫 長期累積的事實，供 RAG 隨時檢索

3. Prompt Engineering（提示工程）：與 AI 溝通的介面

如果說 Token 是 AI 思考的詞彙，Context 是它的記憶體，那麼 Prompt Engineering（提示工程） 就是驅動這台超級電腦的「程式碼」。
AI 模型本質上是一個「文字接龍」機率預測器。提示工程的核心，就是透過精確的文字結構與邏輯引導，將大模型原本發散、隨機的輸出，限縮並聚焦在我們需要的特定任務上。
以下為您拆解進階提示工程的結構、核心策略與底層邏輯：

3.1. 標準 Prompt 的四大結構

一個能讓 AI 穩定輸出高質量答案的專業 Prompt，通常包含以下四個核心要素：

  • 角色與背景 (Role & Context)：賦予 AI 特定身份（例如：「你是一位擁有 10 年經驗的資深資訊安全審查員」）。這會改變模型內部的權重權衡，使其調用特定領域的 Token 語料。
  • 具體任務 (Task/Instruction)：用明確的動詞下達核心指令（例如：「請審查以下程式碼並找出潛在的 SQL 注入漏洞」）。
  • 約束與格式 (Constraints & Output Format)：嚴格限制邊界與輸出長相（例如：「僅輸出 JSON 格式」、「若無法確定請回答不知道，不可瞎編」）。
  • 輸入資料 (Data/Input)：需要處理的文本、程式碼或數據，通常用 XML 標籤（如 ）包覆起來，防止 AI 產生指令注入混淆。

3.2. 三大進階推理策略（讓 AI 學會「思考」）

為了處理複雜的邏輯推理，單純直接提問容易導致 AI 產生幻覺（胡言亂語）。以下是目前業界最核心的提示策略：

① CoT (Chain of Thought，思考鏈)

  • 作法：在指令中加入「請一步一步思考 (Let’s think step by step)」，或在範例中寫出解題的推理步驟。
  • 底層原理：AI 生成下一個 Token 時，必須依賴前面已生成的 Token。如果強迫它先寫下推理過程，這些「中間推理 Token」就會成為它計算最終答案的正確上下文（Context），從而將複雜數學或邏輯的準確率提升數倍。

② Few-Shot Prompting (少樣本學習)

  • 作法：在 Prompt 中直接提供 2~3 個「輸入與標準輸出」的範例（Examples），再給出真正的題目。
  • 底層原理：這是在不重新訓練模型（Fine-tuning）的情況下，利用 Context Window 進行「情境內學習 (In-Context Learning)」，讓 AI 即時抓到你想要的格式、語氣與邏輯複雜度。

③ ReAct (Reasoning + Acting，推理與行動)

  • 作法：這是走向 AI Agent 的關鍵。引導 AI 以「思考 (Thought) – > 行動 (Action) – > 觀察 (Observation)」的循環來撰寫 Prompt。
  • 範例：
  • Thought: 我需要知道目前的氣溫，但我不知道時間，我應該查詢網路。
    • Action: google_search(“台北今日天氣”)
    • Observation: 台北天氣為 28 度，晴天。
    • Thought: 得到數據了，我可以回答使用者。

3.3. 2026 年提示工程的演變：從「寫作文」到「結構化程式碼」

隨著大模型能力提升，現代提示工程已經不再只是單純的「跟 AI 聊天」，而是全面走向結構化與偽程式碼化：

  • XML 標籤隔離：使用 、、、 等標籤。這能讓當前具備強大結構理解能力的模型（如 Claude 3.5 / GPT-4o）精確定位資訊，避免指令與資料混淆。
  • 結構化 Prompt 框架 (如 Markdown/JSON)：將 Prompt 用 Markdown 的標題、清單進行分層設計，甚至使用類似虛擬程式碼（Pseudocode）的邏輯（例如 IF user_input == “help” THEN…）來約束 AI 的行為。

4. AI Agent（智慧代理）：具備自主行動力的 AI

如果說 Token 是細胞、Context 是記憶、Prompt 是思維框架，那麼 AI Agent（智慧代理） 就是真正具備四肢與大腦、能夠在數位世界裡自主行走的「人型生物」。
傳統的 AI 聊天機器人（Chatbot）是「被動」的，你問一句，它答一句。而 AI Agent 是「主動」的，你給它一個最終目標，它會自己思考、自己拆解步驟、自己翻箱倒櫃找工具，直到把任務完成為止。
以下為您深入拆解 AI Agent 的核心架構、底層運作循環以及 2026 年最前沿的架構演進：

4.1. AI Agent 的四大核心支柱

一個完整的 AI Agent 系統，由 OpenAI 首席科學家 Lilian Weng 提出的經典架構定義，主要包含以下四個部分：

  • 大腦 (Core LLM)：作為中央決策系統，負責理解人類意圖、進行邏輯推理。
  • 規劃能力 (Planning)：
    • 任務拆解：將宏觀目標（如「幫我寫一份競品分析報告」）拆解為可執行的小任務。
    • 反思與自我修正：在執行過程中，自我檢查結果是否符合預期，若出錯會自動切換路徑。
  • 記憶系統 (Memory)：
    • 短期記憶：當前任務的執行狀態、Context 視窗內的對話軌跡。
    • 長期記憶：透過向量資料庫或外部檔案，紀錄使用者的長期偏好、過往成功經驗。
  • 工具調用 (Tools/Action)：賦予 AI 改變外部世界的能力（如呼叫 Google API 搜尋、執行 Python 程式碼、發送 Slack 訊息、串接資料庫）。

4.2. 底層核心運作循環：ReAct 模式

Agent 是如何一邊思考一邊做事的？業界最常用的核心機制是 ReAct (Reasoning + Acting) 循環。
當你對 Agent 下達指令：「幫我查美金匯率，並將 5000 美金換算成台幣。」
它的內心小劇場（底層 Log）會這樣運作：

  1. Thought（思考）：使用者想要轉換 5000 美金到台幣。我需要先知道目前最新的美金對台幣匯率。我應該使用匯率查詢工具。
  2. Action（行動）：呼叫工具 get_exchange_rate(from=”USD”, to=”TWD”)。
  3. Observation（觀察/工具回傳結果）：工具回傳 32.54。
  4. Thought（思考）：得到匯率為 32.54。現在我需要計算 5000 * 32.54。我應該呼叫計算機工具。
  5. Action（行動）：呼叫工具 calculator(“5000 * 32.54”)。
  6. Observation（觀察）：工具回傳 162700。
  7. Thought（思考）：計算完成，結果是 162,700 元台幣。我可以回答使用者了。
  8. Final Answer（最終回答）：為您查詢完畢，目前美金兌台幣匯率為 32.54，5,000 美金折合台幣約為 162,700 元。

4.3. Multi-Agent（多智慧代理協同）：當前的最前沿架構

進入 2026 年，單一 Agent（Solo Agent）處理複雜任務的瓶頸越來越明顯（容易在過長的思考鏈中迷失）。目前的趨勢是 Multi-Agent 架構——將任務分工給一整個 AI 團隊：

  • PM Agent（專案經理）：負責接收人類任務，拆解並分配給不同的專業 Agent，最後整合成果。
  • Coder Agent（工程師）：專職編寫程式碼，不參與其他雜務。
  • Tester Agent（測試員）：專門執行 Coder 寫好的程式碼，找出 Bug 並丟回給 Coder 修復。
  • 架構框架：目前主流的開發框架如 LangGraph、AutoGen 與 CrewAI，都是圍繞著「如何讓多個 AI 像人類公司一樣開會、分工、協同作業」來設計的。

4.4. 總結：從 Token 到 Agent 的完整生命週期

我們用一個簡單的例子，把這四個概念全部串起來：
當你對一個 AI 旅遊 Agent 說：「幫我規劃明天的台北一日遊。」

  1. 系統將你的話切成 Tokens，透過 Embedding 變成數學向量。
  2. Prompt Engineering 規範了這個 AI 必須扮演「專業導遊」，且必須以 JSON 格式輸出行程。
  3. AI 從 Context（工作記憶） 發現不知道明天天氣，於是去 向量資料庫（長期記憶） 撈出你曾說過「討厭下雨天」的紀錄。
  4. Agent 大腦開始規劃：先呼叫「氣象 API（工具）」，發現明天會下雨；於是它自我反思修正，放棄陽明山行程，改呼叫「Google Maps API（工具）」搜尋台北室內美術館與咖啡廳。
  5. 最後，Agent 產出了一份完美的雨天備案行程表給您。

總結比較：從 Chatbot 到 Agent

  • 傳統 Chatbot：等待人類輸入 – > 預測下一個 Token（單純回話）。
  • AI Agent：接收目標 –> 自行拆解任務 – > 查詢工具 – > 執行 – > 驗證 – > 交付結果。

在 VS Code 中操作 GitHub

在 VS Code 中操作 GitHub，最核心的步驟包含安裝工具、連結帳號、下載專案與提交變更。透過 VS Code 內建的「原始碼控制」面板，即可完成圖形化的版本控制，免去記憶複雜的指令。以下為您整理具體的操作步驟與重點。

1. 事前準備在使用 VS Code 操作 GitHub 之前，您需要先確保電腦中已安裝好以下兩項基本工具：

  • 安裝 Git：前往 Git 官方網站 下載並完成安裝（安裝過程一路點選下一步即可）。
  • GitHub 帳號：若沒有帳號，請至 GitHub 官網 註冊。

2. 將 VS Code 連線至 GitHub

  1. 打開 VS Code，點擊左側活動列的「原始碼控制」圖示（圖示為三個點連接的樹狀結構）。

  1. 點擊「登入」或選擇「從 GitHub 複製」(Clone from GitHub)，這時瀏覽器會自動跳轉至 GitHub 授權頁面。

  2. 登入 GitHub 並點擊授權，授權成功後瀏覽器會詢問是否開啟 VS Code，確認開啟即可完成帳號連結。

3. 下載現有專案 (Clone)

若要將 GitHub 上的專案下載到本機電腦進行修改：

  1. 開啟 VS Code，按下快速鍵 Ctrl + Shift + P（Mac 上為 Cmd + Shift + P）叫出「命令選擇區」。
  2. 輸入 gitcl，並選擇 Git: Clone 指令。
  3. 貼上專案的 GitHub 網址，或直接點選「從 GitHub 複製」來選擇您的專案。
  4. 選擇電腦中的一個資料夾來儲存此專案，下載完成後直接在 VS Code 開啟該資料夾。

4. 提交與同步變更 (Commit & Push)

當您在 VS Code 中修改了專案程式碼後，需將變更上傳至 GitHub：

  1. 查看變更：點擊左側「原始碼控制」圖示，您會看到修改過的檔案。
  2. 暫存變更 (Stage)：將游標移到欲提交的檔案上方，點擊右側出現的「＋」號（或點擊全部變更右側的 + 號）。
  3. 提交 (Commit)：在輸入框中填寫本次修改的「提交訊息」（例如：更新首頁版面），接著點擊上方的「✓」勾勾圖示。
  4. 同步 (Push)：點擊面板下方的「同步變更」按鈕，或點擊右上角的「…」選單選擇「推送」(Push)，即可將程式碼同步上傳到 GitHub。

5. 擴充功能推薦

若要獲得更強大的 GitHub 整合功能，建議在 VS Code 安裝以下擴充套件：

  • GitHub Pull Requests and Issues：直接在編輯器內審查 PR、管理 Issues。
  • GitLens — Git supercharged：提供強大的程式碼歷史紀錄、誰在什麼時候修改了哪一行程式碼等詳細資訊。

Git 查看內容常用指令

以下是 Git 中專門用來「查看內容」的常用指令：

1. 查看「檔案內容」與修改者

  • git blame <檔案路徑>：逐行顯示檔案內容，並標註每一行最後是由誰、在選定 Commit 中修改。
  • git show :<檔案路徑>：直接在終端機印出該檔案在特定歷史版本中的完整內容。

2. 查看「檔案差異（Diff）」

  • git diff <檔案路徑>：查看工作區與暫存區（尚未 add 的修改）的內容差異。
  • git diff –cached <檔案路徑>：查看已暫存（已 add 但尚未 commit）的內容差異。
  • git diff <檔案路徑>：比較該檔案在兩個不同版本之間的內容改變。

3. 查看「提交紀錄與變更內容」

  • git log -p <檔案路徑>：顯示該檔案的提交歷史，並逐筆列出每次修改的程式碼細節。
  • git show ：查看某次提交的詳細資訊，包含作者、時間以及該次提交所有檔案的修改內容。
  • git log –stat：檢視歷史紀錄時，只顯示每次提交有哪些檔案被變更，以及各自新增/刪除的行數統計。

4. 查看「特定物件與檔案清單」

  • git cat-file -p ：檢視 Git 內部資料庫中特定 Blob 物件的原始內容。
  • git ls-tree -r <分支名或Commit_ID>：列出特定版本中，專案內所有檔案的清單與其對應的物件 ID。

在 Git 中搜尋檔案內容的特定關鍵字歷史，主要有兩種核心武器：

  • git log -S（搜尋字> 串出現次數的變化）
  • git log -G（使用正規表示式搜尋）。

這能幫您快速找出「某行程式碼」或「某個變數」是在哪一次提交被加進去或刪除的。

5. 核心指令：按關鍵字搜尋歷史變更

🔹 尋找關鍵字「新增」或「刪除」的提交（Pickaxe 搜尋）
這是最常用的指令。它會找出關鍵字「出現次數發生改變」的提交紀錄（例如：從無到有、或整行被刪除）。

  • git log -S “關鍵字” -p

    -p：強烈建議加上，它會直接秀出該次提交的詳細程式碼差異（Diff），讓您一眼看到關鍵字在哪一行。

    🔹 使用正規表示式（Regex）搜尋變更

    如果您想搜尋符合某種規則的文字，或者只要該行有被修改（即使關鍵字總數沒變）就列出來，請使用 -G：

  • git log -G “正規表示式” -p

    範例：git log -G “func.*Auth” -p（尋找包含 func 且後面有 Auth 的程式碼變動）

6. 進階技巧：縮小搜尋範圍

🔹 限定只搜尋「特定檔案」
   如果您已經知道該關鍵字大概在哪個檔案裡，務必在尾端加上檔案路徑，速度會快很多：

  • git log -S “關鍵字” -p – 檔案路徑

    🔹 顯示精簡清單（只看 Commit 摘要）
    如果不想看長長的程式碼差異，只想先列出哪幾次提交有動到這個字：

  • git log -S “關鍵字” –oneline

    🔹 搜尋「所有分支」的歷史
    預設只會搜尋目前所在的分支。若想連同其他分支、過去刪除的分支歷史一起撈，請加上 –all：

  • git log -S “關鍵字” –all -p

💡 額外補充：只想搜尋「目前最新版本」的關鍵字？

如果您不是要找歷史，只是想在目前所有檔案中找出誰用了這個關鍵字，請用 git grep（速度比萬用的 grep 快非常多）：

  • git grep “關鍵字”

  • git grep -n “關鍵字”：連同文字在檔案的「行號」一起顯示。

在 Git 的工作流程中，檔案會經歷三個主要區域：工作區 (Working Directory)、暫存區 (Staging Area / Index) 與 本地庫 (Local Repository / HEAD)。
要比較這三者之間的內容差異，最核心的工具就是 git diff。以下為您整理對應的比較指令與心法：

7. 核心比較指令

🔹 比較「工作區」與「暫存區」

   當您修改了檔案，但還沒有執行 git add 時，想看自己改了什麼：

  • git diff

    意義：查看哪些修改尚未被記錄到暫存區。

    🔹 比較「暫存區」與「本地庫 (HEAD)」

    當您執行了 git add，但還沒有執行 git commit 時，想確認準備要提交的內容：

  • git diff –cached

    (注意：git diff –staged 的效果完全相同，兩者挑一個好記的即可。)
    意義：查看下一次執行 git commit 時，會被送進本地庫的內容。
    🔹 比較「工作區」與「本地庫 (HEAD)」
    想略過暫存區的狀態，直接看「目前目錄下的檔案」和「最後一次 Commit 的版本」有何不同：

  • git diff HEAD

    意義：查看自從上一次提交以來，工作區累積的所有變更（包含已 add 與未 add 的修改）。

8. 進階實用技巧

📌 只查看特定檔案的差異

上述指令預設會比較整個專案。若只想專注在某個檔案，請在指令最後加上檔案路徑：
  • git diff README.md
  • git diff –cached README.md

📌 只想看檔案清單，不想看密密麻麻的程式碼

如果檔案改動很多，只想確認有哪些檔案狀態不同、影響了幾行，可以加上 --stat 參數：
  • git diff –stat
  • git diff –cached –stat

💡 快速總結圖解

git diff ＝ 工作區 🆚 暫存區

git diff --cached ＝ 暫存區 🆚 本地庫 (HEAD)

git diff HEAD ＝ 工作區 🆚 本地庫 (HEAD)

放棄工作區中「尚未暫存（尚未 git add）」的修改，可以依照您的需求範圍（單一檔案、整個專案）選擇以下對應的指令。

執行後，這些檔案的內容將會直接還原到與暫存區（或最後一次提交）相同的狀態。

9. 放棄「特定檔案」的修改

如果您只想還原某個或某些檔案，請執行：

  • git restore <檔案路徑>

範例：git restore README.md

10. 放棄「整個專案（所有檔案）」的修改

如果您想一口氣取消目前所有已修改、但尚未暫存的檔案變更，請執行：

  • git restore .

(注意：點 . 代表當前目錄下的所有檔案)

⚠️ 重要提醒：這會動到「未追蹤的檔案（Untracked Files）」嗎？

上述的 checkout 或 restore 指令只會影響 Git 已經在追蹤的檔案。如果您在工作區新建立了一個全新的檔案，但從未執行過 git add，上述指令不會刪除它。

如果您想連同這些「新建立、未追蹤的檔案/資料夾」也一起清空：
請執行清潔指令：

  • git clean -fd

    • -f (force)：強制刪除。
    • -d：連同未追蹤的整個資料夾一起刪除。

💡 實用備案：不想刪除，只想先「借放」到旁邊？

如果您不確定未來會不會用到這些修改，直接刪除太冒險，建議使用 stash（暫存棧）功能把修改藏起來，等以後需要再拿出來：

  • git stash -u

    -u：連同未追蹤的新檔案一起打包藏起來，讓工作區瞬間變乾淨。

11. 檢查目前工作區是否已經完全回到乾淨狀態

最核心且常用的指令是 git status。請在終端機執行：

  • git status

🔍 如何判斷是否「完全乾淨」？

執行後，如果看到以下關鍵字，代表您的工作區非常乾淨，沒有任何未提交的變更：

nothing to commit, working tree clean

⚠️ 如果畫面「不乾淨」，常見的狀態提示有哪些？

如果執行 git status 後沒有出現上述的 clean 訊息，通常會看到以下幾種狀況：

  1. 工作區有未暫存的修改 (Changes not staged for commit):

    畫面顯示：紅色的檔案名稱。

    代表含意：有些追蹤中的檔案被修改了，但您還沒有執行 git add。

    解決方式：若要放棄，請執行 git restore <檔案>。

  2. 有未追蹤的新檔案 (Untracked files):

    畫面顯示：紅色的檔案名稱，出現在 “Untracked files:” 區塊。

    代表含意：專案中出現了全新建立的檔案，且從未被納入 Git 管理。

    解決方式：若想徹底刪除這些新檔案，請執行 git clean -fd。

  3. 暫存區有東西準備提交 (Changes to be committed):

    畫面顯示：綠色的檔案名稱。

    代表含意：有些修改已經執行了 git add，正等待被 commit。

    解決方式：若想把這些綠色檔案退回工作區，請執行 git restore –staged <檔案>。

💡 進階技巧：精簡版檢查
如果您不想看太長的文章說明，只想快速確認有沒有髒檔案，可以加上 -s 參數：

  • git status -s

如果執行後完全沒有任何輸出內容，就代表工作區絕對乾淨！

介紹

本文檔給出了構成主要 Python 發行版中標準函式庫的 Python 程式碼的編碼約定。請參閱隨附的資訊 PEP，描述Python 的 C 實作中的 C 程式碼的樣式指南。

本文檔和PEP 257（文檔字串約定）改編自 Guido 的原始 Python 樣式指南文章，並添加了 Barry 的樣式指南[2]中的一些內容。

隨著時間的推移，隨著附加約定的確定以及過去的約定因語言本身的變化而變得過時，這種風格指南也在不斷發展。

許多項目都有自己的程式設計風格指南。如果發生任何衝突，則此類特定於專案的指南優先適用於該專案。

盲目的一致追求，是狹隘心靈所寄居的妖怪

Guido 的重要見解之一是，程式碼的讀取次數遠多於編寫次數。此處提供的指南旨在提高程式碼的可讀性並使其在各種 Python 程式碼中保持一致。正如PEP 20所說，「可讀性很重要」。

風格指南是關於一致性的。與本風格指南的一致性很重要。專案內的一致性更為重要。一個模組或功能內的一致性是最重要的。

然而，知道何時要不一致——有時風格指南的建議並不適用。如有疑問，請運用您的最佳判斷。查看其他範例並決定哪個看起來最好。請隨時詢問！

特別是：不要僅僅為了遵守此 PEP 而破壞向後相容性！

忽略特定準則的其他一些充分理由：

應用指南會降低程式碼的可讀性，即使對於習慣閱讀遵循此 PEP 的程式碼的人也是如此。

因為相關程式碼早於指南的引入，並且沒有其他原因需要修改該程式碼。

最大線路長度

將所有行限制為最多 79個字元。

對於結構限制較少（文件字串或註解）的流動長文字區塊，行長度應限制為 72 個字元。

限制所需的編輯器視窗寬度可以並排開啟多個文件，並且在使用在相鄰列中顯示兩個版本的程式碼審查工具時效果很好。

大多數工具中的預設包裝會破壞程式碼的視覺結構，使其更難以理解。選擇這些限制是為了避免在視窗寬度設定為 80 的編輯器中進行換行，即使工具在換行時在最後一列中放置了標記符號。一些基於網路的工具可能根本不提供動態換行。

有些團隊強烈喜歡更長的線路。對於專門或主要由可以就此問題達成一致的團隊維護的代碼，可以將行長度限制增加到 99 個字符，前提是註釋和文檔字符串仍以 72 個字符換行。

Python 標準函式庫比較保守，要求將行數限制為 79 個字元（文件字串/註解限制為 72 個）。

用兩個空白行包圍頂層函數和類別定義。

類別內的方法定義由一個空白行包圍。

可以（謹慎地）使用額外的空白行來分隔相關函數組。在一堆相關的單行程式碼（例如一組虛擬實作）之間可以省略空白行。

Python 接受 control-L（即 ^L）換頁符號作為空格；許多工具將這些字元視為頁面分隔符，因此您可以使用它們來分隔文件的相關部分的頁面。請注意，某些編輯器和基於 Web 的程式碼檢視器可能無法將 control-L 識別為換頁符，並會在其位置顯示另一個字形。

原始檔編碼

命名約定

Python 函式庫的命名約定有點混亂，因此我們永遠無法使其完全一致——儘管如此，以下是目前推薦的命名標準。新的模組和套件（包括第三方框架）應按照這些標準編寫，但如果現有庫具有不同的風格，則首選內部一致性。

作為 API 公共部分對使用者可見的名稱應遵循反映用法而不是實現的約定。

描述性：命名風格

有很多不同的命名風格。它有助於識別正在使用的命名樣式，而與它們的用途無關。

通常區分以下命名風格：

  • b（單個小寫字母）
  • B（單一大寫字母）
  • lowercase
  • lower\_case\_with\_underscores
  • UPPERCASE
  • UPPER\_CASE\_WITH\_UNDERSCORES
  • CapitalizedWords（或 CapWords，或 CamelCase——如此命名是因為它的字母看起來凹凸不平[4]）。有時也稱為 StudlyCaps。
    注意：在 CapWords 中使用縮寫時，請將縮寫的所有字母大寫。因此 HTTPServerError 比 HttpServerError 更好。

我們在這裡不使用術語“私有”，因為在 Python 中沒有屬性是真正私有的（通常不需要做不必要的工作）。

另一類屬性是「子類 API」的一部分（在其他語言中通常稱為「受保護」）。有些類別被設計為繼承、擴展或修改類別行為的各個方面。在設計這樣的類別時，請注意明確決定哪些屬性是公共的、哪些是子類別 API 的一部分以及哪些確實只能由基底類別使用。

  • 公共屬性不應有前導底線。

    註 1：有關類別方法，請參閱上面的參數名稱建議。
  • 對於簡單的公共資料屬性，最好隻公開屬性名稱，而不需要複雜的存取器/修改器方法。請記住，如果您發現簡單的資料屬性需要擴展功能行為，Python 提供了一條未來增強的簡單路徑。在這種情況下，請使用屬性將功能實作隱藏在簡單的資料屬性存取語法後面。

公共和內部介面

任何向後相容性保證僅適用於公共介面。因此，使用者能夠清楚地區分公共介面和內部介面非常重要。

記錄的接口被認為是公共的，除非文件明確聲明它們是臨時或內部接口，不受通常的向後相容性保證。所有未記錄的介面都應假定為內部介面。

為了更好地支援自省，模組應使用屬性在其公共 API 中明確聲明名稱\_\_all\_\_。設定 \_\_all\_\_為空列表表示該模組沒有公共 API。

即使\_\_all\_\_設定正確，內部介面（套件、模組、類別、函數、屬性或其他名稱）仍應以單一前導底線為前綴。

如果任何包含命名空間（套件、模組或類別）被視為內部的，則介面也被視為內部的。

導入的名稱應始終被視為實作細節。其他模組不得依賴對此類導入名稱的間接訪問，除非它們是包含模組的 API 的明確記錄部分，例如os.path或公開子\_\_init\_\_模組功能的包模組。

  • Remote (遠端): 連結 GitHub/GitLab 帳號後，可直接查看雲端上的專案。
  • Clone (複製): 將雲端倉庫（如 GitHub 網址）下載到本地電腦。
  • Add (加入): 將電腦中現有的 Git 資料夾掛載到 SourceTree 列表中。
  • Create (建立): 在指定的資料夾初始化一個全新的 Git 倉庫。

2. 環境設定與 SSH 配置 (Options & SSH)

為了確保能順利提交代碼並與遠端伺服器連線，請確認以下設定：

Tools > Options

  • 使用者資訊 (Default user information):
    • Full Name: 設定你在 Commit 紀錄中顯示的名字。
    • Email Address: 必須與你的 Git 平台（如 GitHub）帳號信箱一致。
  • SSH 客戶端設定: 建議使用 OpenSSH 並正確指向你的私鑰路徑。

3. 遠端倉庫與金鑰指定 (Remote Settings)

當您需要將專案推送到雲端時，必須設定遠端連結路徑，並指定正確的身份驗證金鑰。

Settins > Remotes

  • Remote name: 預設通常為 origin，這是遠端倉庫的代號。
  • URL / Path: 填寫雲端倉庫的位址。
    • 進階技巧: 如果您有多組 SSH Key，可以在路徑前面使用別名（例如 keyB-deploy:帳號/倉庫.git），這能強制 SourceTree 使用您在 config 檔中定義的特定金鑰與該倉庫通訊。
  • Remote Account: 連結特定的主機帳號（如 GitHub），以便進行更深層的整合操作。

4. 工作區介面解析 (File Status View)

當你開啟一個專案後，最常用的就是「File Status」視圖，這裡處理「現在」的改動。

核心區域說明：

  • 左側導覽列: 切換「檔案狀態」、「歷史紀錄」、「分支管理」。
  • 已暫存區 (Staged files): 準備要打包提交的檔案。
  • 未暫存區 (Unstaged files): 電腦中已被修改但尚未打包的檔案。
  • 比對區 (Diff View): 點擊檔案可看具體改動（紅色代表刪除，綠色代表新增）。

5. 頂部工具列詳解 (Action Toolbar)

這是你對 Git 下指令的「動作按鈕區」，也是最核心的控制中心：

  • Commit (提交): 將已暫存 (Staged) 的檔案「打包」並加上訊息備註，存入你電腦的本地紀錄中。
  • Pull (拉取) / Push (推送): * Pull: 從雲端伺服器下載最新的進度到你的電腦。
    • Push: 把你電腦裡的最新紀錄上傳到雲端伺服器分享給他人。
  • Fetch (擷取): 檢查雲端有沒有新東西，但先不自動合併，讓你先掌握遠端動態。
  • Branch (分支) / Merge (合併): * Branch: 建立「平行時空」或開外線來開發新功能，不影響主線。
    • Merge: 將不同分支的開發成果合併在一起。
  • Stash (暫存): 臨時的隱藏區。如果你程式寫到一半必須去處理緊急 Bug，可以用 Stash 將目前改動暫時「藏」起來。
  • Discard (放棄修改): 點擊後會把你選取的檔案改動全部抹除，還原到最後一次存檔的狀態。
  • Tag (標籤): 為特定的 Commit 加上標籤（如 v1.0），方便日後快速回溯。

6. 歷史紀錄與版本溯源 (History View)

「History」視圖讓你查看專案的「過去」。

  • 分支圖 (Graph): 以線條顯示分支的開叉與合併過程。
  • 標籤指示: main 代表本地位置，origin/main 代表雲端位置。

7. 快速操作工作流

  1. 修改: 在編輯器改完存檔。
  2. 暫存: 在 Unstaged files 點擊 + 移至 Staged files。
  3. 提交: 點擊 Commit 並輸入訊息。
  4. 同步: 點擊 Push 上傳至雲端。
