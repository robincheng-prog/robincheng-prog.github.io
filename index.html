自解壓縮執行檔設計方案

本方案詳述如何將 demo 專案中所有的目錄結構與檔案，封裝成一個單一的自解壓安裝檔 (demo_setup.exe)。在執行時，此安裝檔會完整還原工作區的目錄結構，隨後自動調用執行目標批次檔 bin\DC.bat。

由於系統環境，本方案 100% 採用 Windows 10/11 內建的原生封裝技術（IExpress 與 PowerShell）。

🛠️ 架構設計與運作原理

本方案由四個核心元件協同運作：

  1. build.bat (建置指令)：在開發環境中執行的自動化打包腳本。
  2. payload.zip (壓縮內容)：包含 demo 所有程式碼、組件與目錄層級的壓縮檔。
  3. launch.bat (自解壓啟動器)：封裝在安裝檔內的引導腳本，負責執行期解壓縮與呼叫主程式。
  4. setup.sed (IExpress 指引設定)：供 Windows 原生封裝工具 iexpress.exe 讀取的設定檔。

📦 元件細節設計

1. 建置指令檔 (build.bat)

位於專案根目錄下，執行時會執行以下操作：

  • 清理舊檔：刪除先前的 payload.zip、launch.bat、setup.sed 與 demo_setup.exe。
  • 打包工作區：調用 PowerShell 將專案目錄的所有檔案與子目錄（排除 .git、建置腳本與設計文件本身）壓縮為 payload.zip，確保目錄層級完整保留。
  • 產生啟動器：自動寫入 launch.bat 至根目錄。
  • 產生配置檔：自動寫入 setup.sed 至根目錄。
  • 執行封裝：調用 %SystemRoot%\System32\iexpress.exe /n /q setup.sed，以無互動背景模式產出 demo_setup.exe。
  • 暫存清理：自動刪除過程中的 payload.zip、launch.bat 與 setup.sed，保持專案乾淨。

2. 啟動器批次檔 (launch.bat)

封裝在 EXE 內的關鍵引導腳本，在解壓至臨時目錄後自動執行：

  • 路徑偵測：支援 %APP_DRIVE% 與 %APP_DIR% 環境變數。若未配置，則預設指向標準路徑 C:\scot\dcdat（此為 DC.bat 預期執行與關聯檔案的正確系統路徑）。
  • 目錄還原：調用 PowerShell 內建的 Expand-Archive 指令，將 payload.zip 解壓縮還原至目標路徑，覆蓋並還原完整的資料夾層級。
  • 主程式調用：切換目錄至 C:\scot\dcdat 並呼叫執行 bin\DC.bat。

PEP 8 – Python 程式碼風格指南

作者： Robin/TJ

介紹

本文檔給出了構成主要 Python 發行版中標準函式庫的 Python 程式碼的編碼約定。請參閱隨附的資訊 PEP，描述Python 的 C 實作中的 C 程式碼的樣式指南。

本文檔和PEP 257（文檔字串約定）改編自 Guido 的原始 Python 樣式指南文章，並添加了 Barry 的樣式指南[2]中的一些內容。

隨著時間的推移，隨著附加約定的確定以及過去的約定因語言本身的變化而變得過時，這種風格指南也在不斷發展。

許多項目都有自己的程式設計風格指南。如果發生任何衝突，則此類特定於專案的指南優先適用於該專案。

盲目的一致追求，是狹隘心靈所寄居的妖怪

Guido 的重要見解之一是，程式碼的讀取次數遠多於編寫次數。此處提供的指南旨在提高程式碼的可讀性並使其在各種 Python 程式碼中保持一致。正如PEP 20所說，「可讀性很重要」。

風格指南是關於一致性的。與本風格指南的一致性很重要。專案內的一致性更為重要。一個模組或功能內的一致性是最重要的。

然而，知道何時要不一致——有時風格指南的建議並不適用。如有疑問，請運用您的最佳判斷。查看其他範例並決定哪個看起來最好。請隨時詢問！

特別是：不要僅僅為了遵守此 PEP 而破壞向後相容性！

忽略特定準則的其他一些充分理由：

應用指南會降低程式碼的可讀性，即使對於習慣閱讀遵循此 PEP 的程式碼的人也是如此。
與周圍也破壞它的程式碼保持一致（可能是出於歷史原因）——儘管這也是清理別人混亂的機會（以真正的 XP 風格）。
因為相關程式碼早於指南的引入，並且沒有其他原因需要修改該程式碼。
當程式碼需要與不支援樣式指南推薦的功能的舊版 Python 保持相容時。

製表符還是空格？

空格是首選的縮排方法。

製表符應僅用於與已使用製表符縮排的代碼保持一致。

Python 不允許混合製表符和空格進行縮排。

最大線路長度

將所有行限制為最多 79個字元。

對於結構限制較少（文件字串或註解）的流動長文字區塊，行長度應限制為 72 個字元。

限制所需的編輯器視窗寬度可以並排開啟多個文件，並且在使用在相鄰列中顯示兩個版本的程式碼審查工具時效果很好。

大多數工具中的預設包裝會破壞程式碼的視覺結構，使其更難以理解。選擇這些限制是為了避免在視窗寬度設定為 80 的編輯器中進行換行，即使工具在換行時在最後一列中放置了標記符號。一些基於網路的工具可能根本不提供動態換行。

有些團隊強烈喜歡更長的線路。對於專門或主要由可以就此問題達成一致的團隊維護的代碼，可以將行長度限制增加到 99 個字符，前提是註釋和文檔字符串仍以 72 個字符換行。

Python 標準函式庫比較保守，要求將行數限制為 79 個字元（文件字串/註解限制為 72 個）。

應該在二元運算子之前還是之後換行？

幾十年來，推薦的風格是在二元運算子之後中斷。但這會以兩種方式損害可讀性：運算符往往會分散在螢幕上的不同列中，並且每個運算符都會從其運算元移到上一行。在這裡，眼睛必須做額外的工作來分辨哪些項目被添加，哪些項目被減去：

為了解決這個可讀性問題，數學家和他們的出版商遵循相反的慣例。 Donald Knuth 在他的電腦和排版系列中解釋了傳統規則：「雖然段落中的公式總是在二元運算和關係之後中斷，但顯示的公式總是在二元運算之前中斷」[3]。

遵循數學傳統通常會產生更具可讀性的程式碼：

在 Python 程式碼中，只要約定在本地一致，就允許在二元運算子之前或之後中斷。對於新程式碼，建議採用 Knuth 風格。

空行

用兩個空白行包圍頂層函數和類別定義。

類別內的方法定義由一個空白行包圍。

可以（謹慎地）使用額外的空白行來分隔相關函數組。在一堆相關的單行程式碼（例如一組虛擬實作）之間可以省略空白行。

在函數中謹慎使用空白行來指示邏輯部分。

Python 接受 control-L（即 ^L）換頁符號作為空格；許多工具將這些字元視為頁面分隔符，因此您可以使用它們來分隔文件的相關部分的頁面。請注意，某些編輯器和基於 Web 的程式碼檢視器可能無法將 control-L 識別為換頁符，並會在其位置顯示另一個字形。

原始檔編碼

核心 Python 發行版中的程式碼應始終使用 UTF-8，且不應具有編碼聲明。

在標準庫中，非 UTF-8 編碼只能用於測試目的。謹慎使用非 ASCII 字符，最好僅用於表示地點和人名。如果使用非 ASCII 字元作為數據，請避免使用雜訊的 Unicode 字符，例如 z̯̯͡a̧͎̺l̡͓̫g̹̲o̡̼̘ 和位元組順序標記。

Python 標準庫中的所有標識符必須使用純 ASCII 標識符，並且應盡可能使用英語單字（在許多情況下，使用非英語的縮寫和技術術語）。

Python 函式庫的命名約定有點混亂，因此我們永遠無法使其完全一致——儘管如此，以下是目前推薦的命名標準。新的模組和套件（包括第三方框架）應按照這些標準編寫，但如果現有庫具有不同的風格，則首選內部一致性。

壓倒一切的原則

作為 API 公共部分對使用者可見的名稱應遵循反映用法而不是實現的約定。

描述性：命名風格

有很多不同的命名風格。它有助於識別正在使用的命名樣式，而與它們的用途無關。

通常區分以下命名風格：

  • b（單個小寫字母）
  • B（單一大寫字母）
  • lowercase
  • lower\_case\_with\_underscores
  • UPPERCASE
  • UPPER\_CASE\_WITH\_UNDERSCORES
  • CapitalizedWords（或 CapWords，或 CamelCase——如此命名是因為它的字母看起來凹凸不平[4]）。有時也稱為 StudlyCaps。
    注意：在 CapWords 中使用縮寫時，請將縮寫的所有字母大寫。因此 HTTPServerError 比 HttpServerError 更好。
  • mixedCase（與 CatalizedWords 的區別在於首字母小寫字元！）
  • Capitalized\_Words\_With\_Underscores（醜的！）
    還有一種使用簡短的唯一前綴將相關名稱分組在一起的風格。這在 Python 中用得不多，但為了完整性還是提到了。例如，該os.stat()函數傳回一個元組，其項目傳統上具有諸如st\_mode、 st\_size等名稱st\_mtime。 （這樣做是為了強調與 POSIX 系統調用結構體字段的對應關係，這有助於程式設計師熟悉這一點。）

X11 函式庫在其所有公用函數中使用前導 X。在Python中，這種風格通常被認為是不必要的，因為屬性和方法名稱以物件為前綴，函數名稱以模組名稱為前綴。

規定性：命名約定

應避免使用的名字

切勿使用字元“l”（小寫字母 el）、“O”（大寫字母 oh）或“I”（大寫字母 eye）作為單字元變數名稱。

在某些字體中，這些字元與數字 1 和 0 無法區分。想要使用“l”時，請改用“L”。

始終決定類別的方法和實例變數（統稱為“屬性”）應該是公共的還是非公共的。如有疑問，請選擇非公開；稍後將其公開比將公共屬性設為非公開更容易。

公共屬性是您期望與您的類別無關的客戶端使用的屬性，並承諾避免向後不相容的變更。非公共屬性是指那些不打算被第三方使用的屬性；您不保證非公共屬性不會更改甚至被刪除。

我們在這裡不使用術語“私有”，因為在 Python 中沒有屬性是真正私有的（通常不需要做不必要的工作）。

另一類屬性是「子類 API」的一部分（在其他語言中通常稱為「受保護」）。有些類別被設計為繼承、擴展或修改類別行為的各個方面。在設計這樣的類別時，請注意明確決定哪些屬性是公共的、哪些是子類別 API 的一部分以及哪些確實只能由基底類別使用。

  • 公共屬性不應有前導底線。

    註 1：有關類別方法，請參閱上面的參數名稱建議。
  • 對於簡單的公共資料屬性，最好隻公開屬性名稱，而不需要複雜的存取器/修改器方法。請記住，如果您發現簡單的資料屬性需要擴展功能行為，Python 提供了一條未來增強的簡單路徑。在這種情況下，請使用屬性將功能實作隱藏在簡單的資料屬性存取語法後面。

公共和內部介面

任何向後相容性保證僅適用於公共介面。因此，使用者能夠清楚地區分公共介面和內部介面非常重要。

記錄的接口被認為是公共的，除非文件明確聲明它們是臨時或內部接口，不受通常的向後相容性保證。所有未記錄的介面都應假定為內部介面。

為了更好地支援自省，模組應使用屬性在其公共 API 中明確聲明名稱\_\_all\_\_。設定 \_\_all\_\_為空列表表示該模組沒有公共 API。

即使\_\_all\_\_設定正確，內部介面（套件、模組、類別、函數、屬性或其他名稱）仍應以單一前導底線為前綴。

如果任何包含命名空間（套件、模組或類別）被視為內部的，則介面也被視為內部的。

導入的名稱應始終被視為實作細節。其他模組不得依賴對此類導入名稱的間接訪問，除非它們是包含模組的 API 的明確記錄部分，例如os.path或公開子\_\_init\_\_模組功能的包模組。

本文件旨在幫助初學者快速上手 SourceTree 的介面與基本功能。SourceTree 是一款強大的 Git 圖形化介面工具，能讓你更直觀地管理程式碼版本。

1. 啟動與倉庫管理 (Repository Management)

在 SourceTree 的初始分頁中，你可以管理你的所有專案。

  • Local (本地): 顯示已存在於你電腦硬碟中的 Git 專案列表。
  • Remote (遠端): 連結 GitHub/GitLab 帳號後，可直接查看雲端上的專案。
  • Clone (複製): 將雲端倉庫（如 GitHub 網址）下載到本地電腦。
  • Add (加入): 將電腦中現有的 Git 資料夾掛載到 SourceTree 列表中。
  • Create (建立): 在指定的資料夾初始化一個全新的 Git 倉庫。

2. 環境設定與 SSH 配置 (Options & SSH)

為了確保能順利提交代碼並與遠端伺服器連線，請確認以下設定：

  • 使用者資訊 (Default user information):
    • Full Name: 設定你在 Commit 紀錄中顯示的名字。
    • Email Address: 必須與你的 Git 平台（如 GitHub）帳號信箱一致。
  • SSH 客戶端設定: 建議使用 OpenSSH 並正確指向你的私鑰路徑。

3. 遠端倉庫與金鑰指定 (Remote Settings)

當您需要將專案推送到雲端時，必須設定遠端連結路徑，並指定正確的身份驗證金鑰。

  • Remote name: 預設通常為 origin，這是遠端倉庫的代號。
  • URL / Path: 填寫雲端倉庫的位址。
    • 進階技巧: 如果您有多組 SSH Key，可以在路徑前面使用別名（例如 keyB-deploy:帳號/倉庫.git），這能強制 SourceTree 使用您在 config 檔中定義的特定金鑰與該倉庫通訊。
  • Remote Account: 連結特定的主機帳號（如 GitHub），以便進行更深層的整合操作。

4. 工作區介面解析 (File Status View)

當你開啟一個專案後，最常用的就是「File Status」視圖，這裡處理「現在」的改動。

核心區域說明：

  • 左側導覽列: 切換「檔案狀態」、「歷史紀錄」、「分支管理」。
  • 已暫存區 (Staged files): 準備要打包提交的檔案。
  • 未暫存區 (Unstaged files): 電腦中已被修改但尚未打包的檔案。
  • 比對區 (Diff View): 點擊檔案可看具體改動（紅色代表刪除，綠色代表新增）。

5. 頂部工具列詳解 (Action Toolbar)

這是你對 Git 下指令的「動作按鈕區」，也是最核心的控制中心：

  • Commit (提交): 將已暫存 (Staged) 的檔案「打包」並加上訊息備註，存入你電腦的本地紀錄中。
  • Pull (拉取) / Push (推送): * Pull: 從雲端伺服器下載最新的進度到你的電腦。
    • Push: 把你電腦裡的最新紀錄上傳到雲端伺服器分享給他人。
  • Fetch (擷取): 檢查雲端有沒有新東西，但先不自動合併，讓你先掌握遠端動態。
  • Branch (分支) / Merge (合併): * Branch: 建立「平行時空」或開外線來開發新功能，不影響主線。
    • Merge: 將不同分支的開發成果合併在一起。
  • Stash (暫存): 臨時的隱藏區。如果你程式寫到一半必須去處理緊急 Bug，可以用 Stash 將目前改動暫時「藏」起來。
  • Discard (放棄修改): 點擊後會把你選取的檔案改動全部抹除，還原到最後一次存檔的狀態。
  • Tag (標籤): 為特定的 Commit 加上標籤（如 v1.0），方便日後快速回溯。

6. 歷史紀錄與版本溯源 (History View)

「History」視圖讓你查看專案的「過去」。

  • 分支圖 (Graph): 以線條顯示分支的開叉與合併過程。
  • 標籤指示: main 代表本地位置，origin/main 代表雲端位置。

7. 快速操作工作流

  1. 修改: 在編輯器改完存檔。
  2. 暫存: 在 Unstaged files 點擊 + 移至 Staged files。
  3. 提交: 點擊 Commit 並輸入訊息。
  4. 同步: 點擊 Push 上傳至雲端。

在標準 Markdown 中，單字內的底線（如 foo_bar_baz）可能會被誤判為斜體。GFM 修正了這個問題，只有當底線兩旁有空格時才會觸發斜體，保護了程式碼變數命名的完整性。

GFM 對單純的換行處理更為直覺。在 GitHub 的評論區中，直接按下 Enter 鍵換行即可換行，不需要像傳統 Markdown 一樣在行末加上兩個空白鍵。

README.md 範本說明

README.md 範本要如何勾勒出了一份專業、易讀且結構完整的 README 檔核心要素。

如果要寫出一份「README.md 內容說明（或撰寫指南）」，一份合格且讓人想讀的 README 應該包含以下五大核心維度與細分項目：

📄 核心維度一：專案門面與概述 (Header & Overview)

這部分是讀者點進專案後的前 3 秒印象，必須精準告訴別人「這是什麼」。

  • 專案標題與標語 (Title & Tagline)： 醒目的專案名稱，並用一句話（或主標題下方的引言）點出核心功能。
  • 狀態徽章 (Badges)： 放置 GitHub Stars、Forks、License、npm 版本、測試覆蓋率等徽章，增加專案公信力。
  • 專案封面圖/Logo (Logo & Banner)： 放置產品 Logo 或視覺封面，提升視覺質感。
  • 快速導覽連結 (Quick Links)： 如線上展示（Demo）連結、官方文件連結、問題回報（Issues）連結。
  • 目錄 (Table of Contents)： 當 README 內容較長時，提供錨點目錄，方便讀者快速跳轉。

🚀 核心維度二：功能展示與操作 (Features & Demo)

讓使用者在下載前，就能直觀感受專案能做到什麼、畫面長怎樣。

  • 動態/靜態畫面展示 (Demo / Screenshots)： 運用 GIF 動畫、操作截圖（1~3 張），直觀呈現產品介面或 CLI 互動過程。
  • 功能清單 (Features Check-list)： 使用 Markdown 的核取方塊（- [x]）條列已完成或支援的核心功能。
  • 測試帳號資訊 (Test Credentials)： 若專案包含登入系統，提供一組「僅供檢視」的測試帳密，方便他人體驗。

🛠️ 核心維度三：開發者指南與安裝步步來 (Getting Started & Usage)

這部分是給想在本機端跑起專案的開發者看的，必須確保步驟清晰無誤。

  • 環境與配置要求 (Prerequisites)： 明確列出限制，例如 Node.js 最低版本需求、需要哪些全域工具（如 npx）或 API Key。
  • 安裝步驟 (Installation)： 從 git clone、移動資料夾、到安裝套件（如 npm install），提供一行行的終端機複製指令。
  • 環境變數說明 (Environment Variables)： 說明 .env 檔案的複製與設定方式，並條列各個變數的用途。
  • 運行專案與開啟 (Usage & Running)： 提供啟動開發環境的指令（如 npm run serve），並標明本地瀏覽器對應的網址（如 http://localhost:8080/）。

📐 核心維度四：專案架構與技術棧 (Architecture & Tech Stack)

幫助想深入研究原始碼、甚至參與貢獻的工程師快速理解專案內層。

  • 資料夾/檔案結構說明 (Project Structure)： 用樹狀圖（Tree）簡述核心資料夾（如 views、controllers、assets）的職責分工。
  • 架構與設計模式 (Architecture)： 引導讀者查閱更詳細的架構文件（如 ARCHITECTURE.md）。
  • 專案技術棧 (Tech Stack & Tools)： 清晰羅列專案所使用的語言、核心框架（如 Vue、Vite）與關鍵第三方服務（如 Google Analytics）。
  • CI/CD 自動化流程說明 (Continuous Integration / Deployment)： 說明專案在發起 Pull Request 或 Merge 到特定分支時，會自動觸發的行為（如：環境建立、ESLint 掃描、測試、部署至 GitHub Pages）。

🤝 核心維度五：社群、版權與聯絡方式 (Community & Footer)

開源專案永續經營或個人職涯曝光的關鍵收尾。

  • 貢獻指南 (Contributing)： 說明如何參與開源，提供 Fork、建立分支、Commit 到 Open PR 的標準工作流。
  • 貢獻者與謝誌 (Contributors & Acknowledgements)： 展示程式碼貢獻者或財務贊助者的頭像/Logo，並致謝啟發本專案的其他開源工具、設計或資源。
  • 作者與聯絡方式 (Author & Contact)： 提供作者的社群連結（如 Twitter、GitHub、LinkedIn、部落格或 Email），建立信任感與個人品牌。
  • 版權與授權說明 (License)： 聲明專案採用的開源協議（通常為 MIT 授權），保障雙方權益。

💡 給您的撰寫建議

在向他人說明 README 該寫什麼時，可以強調「因地制宜」。如果只是一個小前端作品，重點應該放在「功能畫面、線上連結、技術棧、安裝運行」；如果是一個開源工具/套件，重點則要轉移到「動態 Demo、使用參數（CLI 語法）、貢獻指南與 License」。
