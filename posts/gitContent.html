Git 查看內容常用指令

以下是 Git 中專門用來「查看內容」的常用指令：

1. 查看「檔案內容」與修改者

  • git blame <檔案路徑>：逐行顯示檔案內容，並標註每一行最後是由誰、在選定 Commit 中修改。
  • git show :<檔案路徑>：直接在終端機印出該檔案在特定歷史版本中的完整內容。

2. 查看「檔案差異（Diff）」

  • git diff <檔案路徑>：查看工作區與暫存區（尚未 add 的修改）的內容差異。
  • git diff –cached <檔案路徑>：查看已暫存（已 add 但尚未 commit）的內容差異。
  • git diff <檔案路徑>：比較該檔案在兩個不同版本之間的內容改變。

3. 查看「提交紀錄與變更內容」

  • git log -p <檔案路徑>：顯示該檔案的提交歷史，並逐筆列出每次修改的程式碼細節。
  • git show ：查看某次提交的詳細資訊，包含作者、時間以及該次提交所有檔案的修改內容。
  • git log –stat：檢視歷史紀錄時，只顯示每次提交有哪些檔案被變更，以及各自新增/刪除的行數統計。

4. 查看「特定物件與檔案清單」

  • git cat-file -p ：檢視 Git 內部資料庫中特定 Blob 物件的原始內容。
  • git ls-tree -r <分支名或Commit_ID>：列出特定版本中，專案內所有檔案的清單與其對應的物件 ID。

在 Git 中搜尋檔案內容的特定關鍵字歷史，主要有兩種核心武器：git log -S（搜尋字> 串出現次數的變化）與 git log -G（使用正規表示式搜尋）。
這能幫您快速找出「某行程式碼」或「某個變數」是在哪一次提交被加進去或刪除的。

5. 核心指令：按關鍵字搜尋歷史變更

🔹 尋找關鍵字「新增」或「刪除」的提交（Pickaxe 搜尋）
這是最常用的指令。它會找出關鍵字「出現次數發生改變」的提交紀錄（例如：從無到有、或整行被刪除）。

  • git log -S “關鍵字” -p

    -p：強烈建議加上，它會直接秀出該次提交的詳細程式碼差異（Diff），讓您一眼看到關鍵字在哪一行。

    🔹 使用正規表示式（Regex）搜尋變更

    如果您想搜尋符合某種規則的文字，或者只要該行有被修改（即使關鍵字總數沒變）就列出來，請使用 -G：

  • git log -G “正規表示式” -p

    範例：git log -G “func.*Auth” -p（尋找包含 func 且後面有 Auth 的程式碼變動）

6. 進階技巧：縮小搜尋範圍

🔹 限定只搜尋「特定檔案」
   如果您已經知道該關鍵字大概在哪個檔案裡，務必在尾端加上檔案路徑，速度會快很多：

  • git log -S “關鍵字” -p – 檔案路徑

    🔹 顯示精簡清單（只看 Commit 摘要）
    如果不想看長長的程式碼差異，只想先列出哪幾次提交有動到這個字：

  • git log -S “關鍵字” –oneline

    🔹 搜尋「所有分支」的歷史
    預設只會搜尋目前所在的分支。若想連同其他分支、過去刪除的分支歷史一起撈，請加上 –all：

  • git log -S “關鍵字” –all -p

💡 額外補充：只想搜尋「目前最新版本」的關鍵字？

如果您不是要找歷史，只是想在目前所有檔案中找出誰用了這個關鍵字，請用 git grep（速度比萬用的 grep 快非常多）：

  • git grep “關鍵字”

  • git grep -n “關鍵字”：連同文字在檔案的「行號」一起顯示。

在 Git 的工作流程中，檔案會經歷三個主要區域：工作區 (Working Directory)、暫存區 (Staging Area / Index) 與 本地庫 (Local Repository / HEAD)。
要比較這三者之間的內容差異，最核心的工具就是 git diff。以下為您整理對應的比較指令與心法：

7. 核心比較指令

🔹 比較「工作區」與「暫存區」

   當您修改了檔案，但還沒有執行 git add 時，想看自己改了什麼：

  • git diff

    意義：查看哪些修改尚未被記錄到暫存區。

    🔹 比較「暫存區」與「本地庫 (HEAD)」

    當您執行了 git add，但還沒有執行 git commit 時，想確認準備要提交的內容：

  • git diff –cached

    (注意：git diff –staged 的效果完全相同，兩者挑一個好記的即可。)
    意義：查看下一次執行 git commit 時，會被送進本地庫的內容。
    🔹 比較「工作區」與「本地庫 (HEAD)」
    想略過暫存區的狀態，直接看「目前目錄下的檔案」和「最後一次 Commit 的版本」有何不同：

  • git diff HEAD

    意義：查看自從上一次提交以來，工作區累積的所有變更（包含已 add 與未 add 的修改）。

8. 進階實用技巧

📌 只查看特定檔案的差異

上述指令預設會比較整個專案。若只想專注在某個檔案，請在指令最後加上檔案路徑：
  • git diff README.md
  • git diff –cached README.md

📌 只想看檔案清單，不想看密密麻麻的程式碼

如果檔案改動很多，只想確認有哪些檔案狀態不同、影響了幾行，可以加上 --stat 參數：
  • git diff –stat
  • git diff –cached –stat

💡 快速總結圖解

git diff ＝ 工作區 🆚 暫存區

git diff --cached ＝ 暫存區 🆚 本地庫 (HEAD)

git diff HEAD ＝ 工作區 🆚 本地庫 (HEAD)

放棄工作區中「尚未暫存（尚未 git add）」的修改，可以依照您的需求範圍（單一檔案、整個專案）選擇以下對應的指令。

執行後，這些檔案的內容將會直接還原到與暫存區（或最後一次提交）相同的狀態。

9. 放棄「特定檔案」的修改

如果您只想還原某個或某些檔案，請執行：

  • git restore <檔案路徑>

範例：git restore README.md

10. 放棄「整個專案（所有檔案）」的修改

如果您想一口氣取消目前所有已修改、但尚未暫存的檔案變更，請執行：

  • git restore .

(注意：點 . 代表當前目錄下的所有檔案)

⚠️ 重要提醒：這會動到「未追蹤的檔案（Untracked Files）」嗎？

上述的 checkout 或 restore 指令只會影響 Git 已經在追蹤的檔案。如果您在工作區新建立了一個全新的檔案，但從未執行過 git add，上述指令不會刪除它。

如果您想連同這些「新建立、未追蹤的檔案/資料夾」也一起清空：
請執行清潔指令：

  • git clean -fd

    • -f (force)：強制刪除。
    • -d：連同未追蹤的整個資料夾一起刪除。

💡 實用備案：不想刪除，只想先「借放」到旁邊？

如果您不確定未來會不會用到這些修改，直接刪除太冒險，建議使用 stash（暫存棧）功能把修改藏起來，等以後需要再拿出來：

  • git stash -u

    -u：連同未追蹤的新檔案一起打包藏起來，讓工作區瞬間變乾淨。

11. 檢查目前工作區是否已經完全回到乾淨狀態

最核心且常用的指令是 git status。請在終端機執行：

  • git status

🔍 如何判斷是否「完全乾淨」？

執行後，如果看到以下關鍵字，代表您的工作區非常乾淨，沒有任何未提交的變更：

nothing to commit, working tree clean

⚠️ 如果畫面「不乾淨」，常見的狀態提示有哪些？

如果執行 git status 後沒有出現上述的 clean 訊息，通常會看到以下幾種狀況：

  1. 工作區有未暫存的修改 (Changes not staged for commit):

    畫面顯示：紅色的檔案名稱。

    代表含意：有些追蹤中的檔案被修改了，但您還沒有執行 git add。

    解決方式：若要放棄，請執行 git restore <檔案>。

  2. 有未追蹤的新檔案 (Untracked files):

    畫面顯示：紅色的檔案名稱，出現在 “Untracked files:” 區塊。

    代表含意：專案中出現了全新建立的檔案，且從未被納入 Git 管理。

    解決方式：若想徹底刪除這些新檔案，請執行 git clean -fd。

  3. 暫存區有東西準備提交 (Changes to be committed):

    畫面顯示：綠色的檔案名稱。

    代表含意：有些修改已經執行了 git add，正等待被 commit。

    解決方式：若想把這些綠色檔案退回工作區，請執行 git restore –staged <檔案>。

💡 進階技巧：精簡版檢查
如果您不想看太長的文章說明，只想快速確認有沒有髒檔案，可以加上 -s 參數：

  • git status -s

如果執行後完全沒有任何輸出內容，就代表工作區絕對乾淨！
