建立 Issue 操作步驟

在 GitHub 建立 Issue（議題）只需透過專案主頁的 Issues 標籤，即可快速回報 Bug 或提出新功能建議。

作者： TJ

步驟 1：進入專案

首先進入專案頁面。

步驟 2：進入專案議題頁面

點選畫面上方導覽列的 [Issues] 功能標籤。

步驟 3：新增 Issue

進入 Issues 頁面後，點選右側的綠色 [New issue] 按鈕。

步驟 4：填寫議題內容

在建立新議題的頁面中，請依序完成以下欄位填寫：

  • [Add a title]：輸入該 Issue 的標題名稱。
  • [Add a description]：詳細輸入該議題的描述內容。

步驟 5：指派負責人 (Assignees)

在頁面右側的 [Assignees] 區域點選設定圖示（齒輪），於 [Select assignees] 列表中點選欲指派的對象。

步驟 6：設定議題性質 (Labels)

在右側的 [Labels] 區域點選設定圖示（齒輪），依據該議題的性質選擇對應的標籤。

?? 關於 Labels 的說明：
Labels 性質預設已經內建 9 種，也可以依據需求自訂 Labels。目前預設的 Labels 包含：

  • bug：問題（程式出錯）
  • documentation：文件修正或新增
  • duplicate：重複問題
  • enhancement：新功能或請求
  • good first issue：適合新手的簡單議題
  • help wanted：需要額外關注與協助
  • invalid：不正確或無效的議題
  • question：疑問
  • wontfix：不會進行處理或修復

步驟 7：提交並建立 Issue

確認所有欄位與設定皆填寫完成後（填寫範例如下圖所示），點選右下角的綠色 [Create] 或 [Submit new issue] 按鈕送出。

(範例：標題輸入「建立git issue的操作細節粗略」，描述輸入「步驟需要增加更多細節說明」，並掛上 documentation 標籤)

?? 檢視與追蹤狀態

點入可看議題細節和處理狀態。

修改程式碼並進行協作

使用別人的 GitHub 儲存庫（Repository）修改程式碼並進行協作，主要分為兩種管道：一種是你沒有原始專案的直接修改權限（開源專案、公共協作常見），需要透過 Fork 的方式；另一種是對方已經將你加為共同協作者（Collaborator），讓你擁有直接推送（Push）程式碼的權限。
以下為您整理這兩種方式的詳細操作步驟。

管道一：標準 Fork & Pull Request 流程（無直接修改權限）

這是最正統的 GitHub 協作方式。你不會直接修改對方的專案，而是複製一份到自己的帳號下修改，再請求對方合併。

1. Fork 專案

  • 開啟對方的 GitHub 專案網頁。
  • 點擊右上角的 Fork 按鈕。
  • 這會在你的個人帳號下建立一個一模一樣的專案副本。

2. Clone 到本機電腦

  • 在你自己帳號下的該專案頁面，點擊綠色的 Code 按鈕並複製 HTTPS 或 SSH 連結。
  • 開啟本機終端機（Terminal），輸入指令將程式碼下載下來：

3. 建立新分支（Branch）

為了保持程式碼整潔，修改前請務必建立獨立的分支，避免直接在 main 或 master 上修改：

4. 修改程式碼並 Commit

  • 使用程式碼編輯器（如 VS Code）修改檔案。
  • 修改完成後，在終端機存檔並提交：

5. 推送（Push）回你自己的 GitHub

將分支推送到你個人的 GitHub 遠端儲存庫：

git push origin <新分支名稱>

6. 發起 Pull Request (PR)

  • 回到你個人的 GitHub 該專案網頁，系統會偵測到你剛剛推送的分支，並顯示一個綠色的 Compare & pull request 按鈕。
  • 點擊它，填寫你想對原始作者說的話（說明的修改原因與內容）。
  • 點擊 Create pull request 送出。
  • 後續：原始專案的作者會收到通知，如果他檢查沒問題，就會點擊合併（Merge），此時你的程式碼就正式貢獻進去了！

管道二：共同協者者流程（擁有直接修改權限）

如果你們是團隊、朋友或同學一起做專案，對方可以將你加入成為專案成員，此時你不需要進行 Fork。

1. 對方加你為協作者

  • 專案擁有者需要進入該儲存庫的 Settings > Collaborators。
  • 點擊 Add people，輸入你的 GitHub 帳號或 Email 發送邀請。
  • 你必須去信箱收信，或是直接登入 GitHub 接受該專案的邀請。

2. 直接 Clone 原始專案

接受邀請後，你已經有了寫入權限。直接複製對方原始專案的 Code 連結並下載：

3. 建立分支與修改

在團隊協作中，同樣不建議直接修改 main 分支。建議為每個新功能建立分支：

4. 直接 Push 回原始專案

因為你已經是協作者，你可以直接將分支推送到對方的儲存庫：

git push origin <你的分支名稱>

5. 在原始專案發起 PR

到對方的 GitHub 網頁上，直接將你的分支發起 Pull Request，交由團隊成員或負責人 Code Review 後進行 Merge。

💡 實用小技巧：保持程式碼同步

當你在修改程式碼的期間，原始專案可能已經被其他人更新了。為了避免發生衝突（Conflict），在送出你的修改前，可以使用以下方式同步最新進度：

github gfm 說明

GitHub Flavored Markdown (GFM) 是 GitHub 基於標準 Markdown 語法進行擴充與修改的 Markdown 版本。它保留了傳統 Markdown 的易讀性，並增加了許多適合軟體開發者與專案管理使用的強大功能（如表格、工作列表等）。
以下為 GFM 的核心擴充功能與語法說明：

1. 表格 (Tables)

標準 Markdown 原生不支援表格，而 GFM 完美支援。使用豎線 | 分隔欄位，第二行使用破折號 - 區隔表頭，並可用冒號 : 控制對齊方式。

項目名稱 數量 價格
靠左對齊 置中對齊 靠右對齊
蘋果 10 $15

2. 工作列表 (Task Lists)

在無序列表語法後面加上 [ ]（未勾選）或 [x]（已勾選），即可在 GitHub 介面上直接點選或呈現核取方塊。

  • 完成需求分析
  • 撰寫 API 文件
  • 部署測試伺服器

3. 程式碼區塊增強 (Fenced Code Blocks)

GFM 允許使用三個反引號 ``` 包裹多行程式碼，並在開頭指定程式語言，即可開啟語法高亮 (Syntax Highlighting)。此外，還支援直接呈現 Mermaid 圖表。

4. 刪除線 (Strikethrough)

在需要刪除的文字前後加上兩個波浪符號 ~~。

這是一段 被刪除的舊文字 更新後的文字。

5. 自動連結與縮寫引用

  • 網址自動轉換：在 GFM 中，直接輸入網址（如 https://github.com）會自動被轉為可點擊的超連結，不需要使用標準的 文字 語法。
  • Issue 與 PR 引用：輸入 #123 會自動連結到該專案編號 123 的 Issue 或 PR。
  • 使用者提及：輸入 @username 會自動通知並連結到該 GitHub 使用者的個人檔案。
  • Commit 雜湊引用：直接輸入 Commit 的 SHA 雜湊值（例如 a1b2c3d），GitHub 會自動將其轉為該提交的連結。

6. 表情符號支援 (Emoji)

GFM 支援使用前後冒號包圍的關鍵字來直接顯示 Emoji。

這是一個讚 :thumbsup: 和一個愛心 :heart:

7. 忽略底線的單字強調

在標準 Markdown 中，單字內的底線（如 foo_bar_baz）可能會被誤判為斜體。GFM 修正了這個問題，只有當底線兩旁有空格時才會觸發斜體，保護了程式碼變數命名的完整性。

8. 分頁與換行優化

GFM 對單純的換行處理更為直覺。在 GitHub 的評論區中，直接按下 Enter 鍵換行即可換行，不需要像傳統 Markdown 一樣在行末加上兩個空白鍵。

除了前面提到的擴充功能，GitHub Flavored Markdown (GFM) 還支援許多標準 Markdown 的核心語法，以及 GitHub 平台專屬的進階排版指令。
以下為您整理其他在寫 README 或 Issue 時非常常用的語法指令：

1. 摺疊區塊 (Collapsible Details)

當文字或程式碼太長時，可以使用 HTML 的

標籤將內容隱藏，讓頁面更乾淨。讀者點擊箭頭才能展開看詳細內容。

點擊展開看詳細程式碼

2. 警告與提示區塊 (Alerts)

GitHub 官方支援的特殊引用語法，可以呈現出漂亮的彩色提示框。包含五種等級：

[!NOTE]
這是藍色提示框：用於補充說明、背景資訊。

[!TIP]
這是綠色提示框：用於提供更好的做法或建議。

[!IMPORTANT]
這是紫色提示框：使用者必須知道的關鍵資訊。

[!WARNING]
這是黃色提示框：需要立即注意的警告。

[!CAUTION]
這是紅色提示框：可能帶來負面後果或危險的警告。

3. 多媒體嵌與呈現 (Media)

  • 圖片大小控制：預設的 無法調整大小。若要限定寬度，請改用 HTML 標籤：
  • 影片嵌入：直接把 .mp4 或 .mov 影片檔拖曳到 GitHub 的編輯框，它會自動生成支援線上撥放的語法。

4. 顏色標記 (Color Chips)

如果你在撰寫樣式文件，可以用三個反引號包裹語言為 yaml，並輸入十六進位顏色碼，GitHub 會自動在前方顯示該顏色的色塊。

5. 錨點連結 (Anchor Links)

在 GitHub 中，所有的標題（Heading）都會自動被轉為網頁錨點。你可以建立「回到目錄」或跳轉到特定段落的連結：

系統安裝說明點我跳轉到步驟二

步驟二內容…

小提醒：網址部分的標題英文字母必須全部小寫，且空格要變成減號 -（例如：## My Title 要寫成 (#my-title)）。

6. HTML 區塊置中與鍵盤樣式

  • 內容置中：Markdown 預設靠左。如果要置中文字或圖片，可使用 <p align="center">內容</p>。

置中的內容

  • 鍵盤快捷鍵樣式：使用 <kbd> 標籤可以讓文字看起來像鍵盤按鍵。
  • 請按下 Ctrl + C 複製。

如果你想進一步了解更細緻的規範，可以參考 GitHub 官方 GFM 規格說明書。

README.md 範本說明

README.md 範本要如何勾勒出了一份專業、易讀且結構完整的 README 檔核心要素。

如果要寫出一份「README.md 內容說明（或撰寫指南）」，一份合格且讓人想讀的 README 應該包含以下五大核心維度與細分項目：

📄 核心維度一：專案門面與概述 (Header & Overview)

這部分是讀者點進專案後的前 3 秒印象，必須精準告訴別人「這是什麼」。

  • 專案標題與標語 (Title & Tagline)： 醒目的專案名稱，並用一句話（或主標題下方的引言）點出核心功能。
  • 狀態徽章 (Badges)： 放置 GitHub Stars、Forks、License、npm 版本、測試覆蓋率等徽章，增加專案公信力。
  • 專案封面圖/Logo (Logo & Banner)： 放置產品 Logo 或視覺封面，提升視覺質感。
  • 快速導覽連結 (Quick Links)： 如線上展示（Demo）連結、官方文件連結、問題回報（Issues）連結。
  • 目錄 (Table of Contents)： 當 README 內容較長時，提供錨點目錄，方便讀者快速跳轉。

🚀 核心維度二：功能展示與操作 (Features & Demo)

讓使用者在下載前，就能直觀感受專案能做到什麼、畫面長怎樣。

  • 動態/靜態畫面展示 (Demo / Screenshots)： 運用 GIF 動畫、操作截圖（1~3 張），直觀呈現產品介面或 CLI 互動過程。
  • 功能清單 (Features Check-list)： 使用 Markdown 的核取方塊（- [x]）條列已完成或支援的核心功能。
  • 測試帳號資訊 (Test Credentials)： 若專案包含登入系統，提供一組「僅供檢視」的測試帳密，方便他人體驗。

🛠️ 核心維度三：開發者指南與安裝步步來 (Getting Started & Usage)

這部分是給想在本機端跑起專案的開發者看的，必須確保步驟清晰無誤。

  • 環境與配置要求 (Prerequisites)： 明確列出限制，例如 Node.js 最低版本需求、需要哪些全域工具（如 npx）或 API Key。
  • 安裝步驟 (Installation)： 從 git clone、移動資料夾、到安裝套件（如 npm install），提供一行行的終端機複製指令。
  • 環境變數說明 (Environment Variables)： 說明 .env 檔案的複製與設定方式，並條列各個變數的用途。
  • 運行專案與開啟 (Usage & Running)： 提供啟動開發環境的指令（如 npm run serve），並標明本地瀏覽器對應的網址（如 http://localhost:8080/）。

📐 核心維度四：專案架構與技術棧 (Architecture & Tech Stack)

幫助想深入研究原始碼、甚至參與貢獻的工程師快速理解專案內層。

  • 資料夾/檔案結構說明 (Project Structure)： 用樹狀圖（Tree）簡述核心資料夾（如 views、controllers、assets）的職責分工。
  • 架構與設計模式 (Architecture)： 引導讀者查閱更詳細的架構文件（如 ARCHITECTURE.md）。
  • 專案技術棧 (Tech Stack & Tools)： 清晰羅列專案所使用的語言、核心框架（如 Vue、Vite）與關鍵第三方服務（如 Google Analytics）。
  • CI/CD 自動化流程說明 (Continuous Integration / Deployment)： 說明專案在發起 Pull Request 或 Merge 到特定分支時，會自動觸發的行為（如：環境建立、ESLint 掃描、測試、部署至 GitHub Pages）。

🤝 核心維度五：社群、版權與聯絡方式 (Community & Footer)

開源專案永續經營或個人職涯曝光的關鍵收尾。

  • 貢獻指南 (Contributing)： 說明如何參與開源，提供 Fork、建立分支、Commit 到 Open PR 的標準工作流。
  • 貢獻者與謝誌 (Contributors & Acknowledgements)： 展示程式碼貢獻者或財務贊助者的頭像/Logo，並致謝啟發本專案的其他開源工具、設計或資源。
  • 作者與聯絡方式 (Author & Contact)： 提供作者的社群連結（如 Twitter、GitHub、LinkedIn、部落格或 Email），建立信任感與個人品牌。
  • 版權與授權說明 (License)： 聲明專案採用的開源協議（通常為 MIT 授權），保障雙方權益。

💡 給您的撰寫建議

在向他人說明 README 該寫什麼時，可以強調「因地制宜」。如果只是一個小前端作品，重點應該放在「功能畫面、線上連結、技術棧、安裝運行」；如果是一個開源工具/套件，重點則要轉移到「動態 Demo、使用參數（CLI 語法）、貢獻指南與 License」。

以下是一份空白的READM.md的內容，你可以依程式的方向作內容的增減：

了解 GitHub 的 pull request 工作流程

Hello World

按照這個「Hello World」練習，了解 GitHub 的 pull request 工作流程。

簡介

本課程培訓 GitHub 的基本知識，例如儲存庫、分支、提交和拉取請求等。 你將創建自己的Hello World儲存庫並了解 GitHub的拉取請求工作流程，這是建立和查看程式碼的熱門方法。

在本快速入門指南中，您將：

  • 建立和使用儲存庫。
  • 啟動和管理新分支。
  • 對文件進行更改並將其作為提交推送到 GitHub。
  • 開啟並合併拉取要求。

先決條件

  • 你必須擁有 GitHub 帳號。 有關更多信息，請參閱 在 GitHub 上建立帳戶。

  • 你不需要知道如何編碼、使用命令列或安裝 Git（建立 GitHub 的版本控制軟體）。

第 1 步：建立儲存庫

首先我們需要建立一個儲存庫。 儲存庫就好比包含相關項目的資料夾，例如文件、圖像、視頻，甚至其他資料夾。 儲存庫通常會將屬於相同「專案」或正在處理的交易的項目組合在一起。

通常，儲存庫包括一個 README 文件，其中具有項目的相關資訊。 自述文件是用 Markdown 語言編寫的，這是一種易於讀取和編寫的語言，用於設定純文字格式。

GitHub 可在新儲存庫的同時新增 README 檔案。 GitHub 還提供了其他常用選項，例如許可證文件，但你現在不必選擇其中任何一個。

可在 hello-world 儲存庫中儲存靈感和資源，甚至與他人分享和討論。

  1. 在任何頁面的右上角，選擇+，然後按一下「新建儲存庫」。

  1. 在「儲存庫名稱」方塊中，鍵入 hello-world。
  2. 在「說明」方塊中，鍵入簡短說明。例如，鍵入「此儲存庫用於練習 GitHub 流程」。
  3. 選擇儲存庫是「公共」還是「私有」 。
  4. 選擇「新增 README 檔案」。
  5. 按一下「建立儲存庫」。

第 2 步：建立分支

透過分支，您可以同時擁有不同版本的儲存庫。

預設情況下，儲存庫有一個名為 main 的分支，它被視為最終分支。 可在儲存庫中從 main 建立其他分支。

要在不更改主要程式碼來源的情況下向專案添加新功能時，分支將非常實用。 在合併主分支之前，在不同分支上完成的工作不會顯示在主分支上，我們將在本指南的後面部分介紹。 可使用分支進行試驗和編輯，然後再提交至 main。

從 main 分支建立分支時，建立的是 main 在當時的副本或快照。 如果其他人在你處理分支時對 main 分支進行了更改，你可拉取這些更新。

  • main 分支
  • 名為 feature 的新分支
  • feature 在合併到 main 之前，需要經歷「提交更改」、「提交拉取請求」和「討論建議的更改」階段

建立分支

  2. 在檔案清單上方，點選main的下拉式選單。

  1. 在文字方塊中鍵入分支名稱 readme-edits。

  2. 點選「建立分支: 從 main 建立 readme-edits」。 ****

現在你有兩個分支：main 和 readme-edits。 現在，它們看起來完全相同。 接下來，您需要在新的 readme-edits 分支中新增變更。

第 3 步：進行和提交更改

在上一個步驟中建立新分支時，GitHub 會將你轉到作為 readme-edits 副本的新 main 分支的程式碼頁。

您可以對儲存庫中的檔案進行變更並儲存變更。 在 GitHub 上，儲存的變更稱為提交。 每個提交都有一個關聯的提交訊息，該訊息是解釋為什麼進行特定更改的說明。 提交訊息會捕獲您更改的歷史記錄，以便其他參與者可以了解您執行了哪些操作及其原因。

  1. 在你建立的 readme-edits 分支下，點選 README.md 檔案。
  2. 若要編輯文件，請點選 。

  4. 按一下「提交更改」。
  5. 在「提交更改」方塊中，編寫描述更改的提交訊息。
  6. 按一下「提交更改」。

這些變更將僅適用於 readme-edits 分支上的 README 文件，所以此分支現在包含 main 中沒有的內容。

第 4 步：開啟一個拉取請求

現在你已在從 main 建立的分支中進行了更改，接下來可開啟拉取請求。

拉取請求(Pull Request) 是 GitHub 上協作的關鍵。 打開拉取請求後，可以提出更改，要求某人審查和提取您的貢獻並將其合併到其分支中。 拉取請求顯示兩個分支中內容的差異。 變化、增減以不同的顏色顯示。

只要進行提交，便可開啟拉取請求並開始討論，即使在程式碼完成之前亦可。

在這一步驟中，你將在自己的儲存庫中開啟一個拉取請求，然後自行合併。 這是在處理大型專案之前練習 GitHub 流程的好方法。

  1. 按一下儲存庫的 拉取請求 標籤。

  1. 按一下「新建拉取請求」。

  2. 在「範例比較」方塊中，選擇你建立的分支 ****，與 readme-edits（原始分支）進行比較。

  3. 在 Compare（比較）頁面上的差異中查看您的更改，確保它們是您要提交的內容。

  1. 按一下「建立拉取請求」****。

  2. 為拉取請求指定一個標題，並寫下變更的簡要說明。 您可以包含表情符號，並且拖放圖片和GIF。

  3. 按一下「建立拉取要求」****。

審查合併請求

開始與他人協作時，這將是你請求他們進行評審的時間。 這樣就可以在將變更合併到 main 分支之前，讓協作者對拉取請求發表評論或提出更改建議。

第 5 步：合併拉取請求

在最後一步中，你將 readme-edits 分支合併到 main 分支中。 合併拉取請求後，readme-edits 分支上的變更將合併到 main。

有時，拉取請求可能會引入與 main 上的現有程式碼衝突的程式碼變更。 如果有任何衝突，GitHub 將提醒你有關衝突程式碼的信息，並防止合併，直到衝突解決為止。 您可以進行解決衝突的提交，也可以使用拉取請求中的註解與團隊成員討論衝突。

在本演練中，應該沒有任何衝突，因此您已準備好將分支合併到主分支中。

  1. 在拉取請求底部，按一下合併拉取請求以將變更合併到main中。
  2. 按一下「確認合併」。 您將收到一則訊息，指出請求已成功合併且請求已關閉。
  3. 按一下「刪除分支」。 現在你的拉取請求已合併，並且你的更改位於 main 上，接下來你可安全地刪除 readme-edits 分支。 如果要對專案進行更多更改，可以隨時建立新分支並重複此過程。
  4. 按一下返回 **** 儲存庫的「程式碼」hello-world選項卡，以在 main 上查看已發佈的變更。

  • 參數說明：-t ed25519 是目前最推薦的加密算法（比傳統 RSA 更小更快且更安全）；-C 則是備註，通常寫 Email。

  • 過程提示：指令執行後會詢問三個問題，直接按三次 Enter 使用預設路徑且不設置額外密碼即可。

2. 將公鑰新增至 GitHub 帳戶

您必須將「公鑰」交給 GitHub，而「私鑰」則需鎖在自己的電腦裡。

  1. 取得公鑰內容：

  • 檔案位置：C:\Users\你的用戶名.ssh\id_ed25519.pub (副檔名為 .pub 的才是公鑰)。

  • 您可以直接在終端機輸入 cat ~/.ssh/id_ed25519.pub 來顯示並複製內容。

  1. 上傳至 GitHub：

  • 登入 GitHub，點擊右上角頭像 -> Settings。

  • 左側選單找到 SSH and GPG keys。

  • 點擊 New SSH key，在 “Title” 取名（如：Work-PC），並將剛複製的內容貼入 “Key” 區塊中。

如私有倉庫是其他人所有，要請代其代為新增公鑰。
如使用多個倉庫就需要多個SSH Key，可以參考 GitHub 獨立金鑰 or GitHub 獨立金鑰 windows 11
