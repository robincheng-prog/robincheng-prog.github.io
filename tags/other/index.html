自解壓縮執行檔設計方案二

本方案詳述如何將 EDAT 專案中所有的目錄結構與檔案，封裝成一個單一的自解壓安裝檔 (edat_setup.exe)。在執行時，此安裝檔會完整還原工作區的目錄結構，隨後自動調用執行目標批次檔 bin\DC.bat。

由於 IExpress 工具過於老舊，在 modern Windows 10/11 中存在長路徑限制、權限阻擋（需要系統管理員權限編譯與執行）等問題，本方案採用更現代、高相容性且免管理員權限的 Windows 原生 .NET C# 編譯器 (csc.exe) 技術來編譯自解壓安裝檔。

🛠️ 架構設計與運作原理

本方案由三個核心元件協同運作：

  1. build.bat (啟動引導)：在開發環境中執行的入口腳本，負責呼叫 PowerShell 建置核心。
  2. build.ps1 (建置核心)：負責清理舊檔、將工作區打包成 payload.zip、產生 C# 自解壓原始碼並呼叫 csc.exe 編譯成 edat_setup.exe。
  3. payload.zip (壓縮內容)：包含 EDAT 所有程式碼、組件與目錄層級的臨時壓縮檔，會被內嵌至編譯產出的 EXE 資源中。

2. 建置核心腳本 (build.ps1)

使用 PowerShell 編寫，完整建置流程如下：

  • 清理舊檔：清除先前的 payload.zip、Installer.cs 與 edat_setup.exe。
  • 打包工作區：使用 Compress-Archive 將專案內除了 .git、建置腳本與設計文件以外的所有檔案壓縮成 payload.zip。
  • 產生 C# 原始碼：產生 Installer.cs。此程式在執行時會讀取內嵌的 payload.zip 資源，解壓至 %APP_DRIVE%\%APP_DIR%（預設為 C:\scot\dcdat），並執行 bin\DC.bat。
  • 編譯封裝：調用 Windows 內建的 .NET 編譯器 %SystemRoot%\Microsoft.NET\Framework64\v4.0.30319\csc.exe，以 /resource:payload.zip 參數將壓縮檔封裝進執行檔，產出 edat_setup.exe。
  • 暫存清理：自動刪除暫存的 payload.zip 與 Installer.cs。

自解壓縮執行檔設計方案一

以下方案會發生IExpress 編譯失敗（Exit Code 1）的核心原因：

🔍 失敗原因分析

  1. 權限限制 (UAC Block)：在 Modern Windows (Windows 10/11) 中，iexpress.exe 編譯程序以及 MakeCAB 精靈被歸類為高權限管理工具。如果目前的終端機沒有使用「系統管理員身分 (Administrator)」啟動，IExpress 在寫入報告檔案或呼叫 MakeCAB 指引時會被系統無聲阻擋，導致回傳錯誤碼 1。
  2. 無互動 GUI 阻擋：iexpress.exe 是一個老舊的 GUI 應用程式。在無週邊（Headless）或背景服務容器中，一旦發生編譯或權限錯誤，它會試圖跳出 GUI 錯誤提示視窗而卡死或被強制中斷，因此無法在普通終端機下穩定自動化編譯。

💡 突破性解決方案：Windows 原生 C# 編譯器 (csc.exe)

  • 為了徹底擺脫 IExpress 老舊、高權限要求以及不穩定的缺點，我探索了 Windows 10/11 內建的 .NET C# 編譯器 (csc.exe)。
  • 每台 Windows 電腦在安裝 .NET Framework 時都會內建 csc.exe 編譯器（位於 C:\Windows\Microsoft.NET\Framework64\v4.0.30319\csc.exe）。我們可以寫一個極其輕量化的 C# 自解壓啟動程式 Installer.cs，並將工作區的 payload.zip 作為內嵌資源 (Embedded Resource) 編譯進 edat_setup.exe 中！

這樣做有以下巨大優勢：

  • 完全免管理員權限：編譯與生成 edat_setup.exe 不需要任何系統管理員權限。
    100% 原生相容：不依賴 7-Zip 或任何第三方軟體，編譯速度極快且產生的 EXE 非常穩定。
  • 無縫自動化：C# 編譯器是標準的命令列工具，不會像 IExpress 一樣在發生錯誤時彈出 GUI 視窗。
  • 已成功實驗驗證：已在環境中完成小型測試程式的編譯與資源讀取測試，全部順利通過，功能完美運作！

本方案詳述如何將 demo 專案中所有的目錄結構與檔案，封裝成一個單一的自解壓安裝檔 (demo_setup.exe)。在執行時，此安裝檔會完整還原工作區的目錄結構，隨後自動調用執行目標批次檔 bin\DC.bat。

由於系統環境，本方案 100% 採用 Windows 10/11 內建的原生封裝技術（IExpress 與 PowerShell）。

🛠️ 架構設計與運作原理

本方案由四個核心元件協同運作：

  1. build.bat (建置指令)：在開發環境中執行的自動化打包腳本。
  2. payload.zip (壓縮內容)：包含 demo 所有程式碼、組件與目錄層級的壓縮檔。
  3. launch.bat (自解壓啟動器)：封裝在安裝檔內的引導腳本，負責執行期解壓縮與呼叫主程式。
  4. setup.sed (IExpress 指引設定)：供 Windows 原生封裝工具 iexpress.exe 讀取的設定檔。

📦 元件細節設計

1. 建置指令檔 (build.bat)

位於專案根目錄下，執行時會執行以下操作：

  • 清理舊檔：刪除先前的 payload.zip、launch.bat、setup.sed 與 demo_setup.exe。
  • 打包工作區：調用 PowerShell 將專案目錄的所有檔案與子目錄（排除 .git、建置腳本與設計文件本身）壓縮為 payload.zip，確保目錄層級完整保留。
  • 產生啟動器：自動寫入 launch.bat 至根目錄。
  • 產生配置檔：自動寫入 setup.sed 至根目錄。
  • 執行封裝：調用 %SystemRoot%\System32\iexpress.exe /n /q setup.sed，以無互動背景模式產出 demo_setup.exe。
  • 暫存清理：自動刪除過程中的 payload.zip、launch.bat 與 setup.sed，保持專案乾淨。

2. 啟動器批次檔 (launch.bat)

封裝在 EXE 內的關鍵引導腳本，在解壓至臨時目錄後自動執行：

  • 路徑偵測：支援 %APP_DRIVE% 與 %APP_DIR% 環境變數。若未配置，則預設指向標準路徑 C:\scot\dcdat（此為 DC.bat 預期執行與關聯檔案的正確系統路徑）。
  • 目錄還原：調用 PowerShell 內建的 Expand-Archive 指令，將 payload.zip 解壓縮還原至目標路徑，覆蓋並還原完整的資料夾層級。
  • 主程式調用：切換目錄至 C:\scot\dcdat 並呼叫執行 bin\DC.bat。

🧪 驗證與測試計畫

手動測試驗證

  1. 建置測試：
    • 以系統管理員身分執行專案根目錄下的 build.bat。
    • 驗證根目錄下是否成功生成 demo_setup.exe，且無任何編譯報錯。
  2. 安裝與還原測試：
    • 將生成的 demo_setup.exe 放置至乾淨的測試環境中執行。
    • 檢查 C:\demo\ 下是否完整還原了 bin/、config/、devices/ 等所有子資料夾，且檔案大小均正確無誤。
  3. 執行測試：
    • 驗證解壓完畢後，是否已自動開啟終端機並正確執行了 bin\DC.bat 及其內部的所有初始化配置。
